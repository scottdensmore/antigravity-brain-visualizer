The problem: a push only *carries* the transcript when its hash differs from the manifest. A completed session's transcript typically never changes again — and an AI `summary.json` is by nature computed *after* the session finishes. So the common ordering is: final transcript pushed (manifest records hash) → `summary.json` written moments later → no further transcript change → the summary is **stranded permanently**, never syncing. This isn't a rare edge case; for post-hoc summaries it may be the dominant case.

Concrete scenario: run a session to completion, `agent-ingest` pushes the transcript, Antigravity then writes `summary.json`. Every subsequent `agent-ingest` run skips the session (hash unchanged) and the summary never reaches Postgres.

The mtime is already available and `scan.Session` already carries `Summary`. Either factor summary presence/hash into the manifest comparison so a newly-appeared summary forces a re-push, or reword the README to state plainly that a summary appearing after the transcript is final will not sync unless the transcript itself changes. As written the doc implies a sync that in practice won't happen.

### 2. `catch (RuntimeException)` conflates a transient outage with permanently-bad data (nit)
`Ingestor.java:112-121`. `summaries.upsert` funnels *every* failure — malformed jsonb, DB outage, pool exhaustion — through one `StoreUnavailableException` (a `RuntimeException`, mapped elsewhere to 503). The new catch swallows all of them and returns `INGESTED`.

Reporting `INGESTED` is actually correct: `sessions.upsert` ran first on its own autocommit connection, so the session is durably stored regardless. And a genuine outage still surfaces — the *next* batch item's `sessions.upsert` (uncaught) throws and propagates a 503. So this is not a correctness bug.

The residual concern is data loss, not misreporting: a *transient* outage striking exactly in the window between the session upsert and the summary upsert silently drops a *valid* summary, and because of finding #1 that summary is then never retried. Low probability, but the broad catch can't distinguish "drop forever, it's malformed" from "retry, the DB blipped." Acceptable as-is given the autocommit-durability design; noting it because the two failure modes genuinely differ.

### 3. `readSummary` is correct; the coerce-then-validate order is right (no action)
`cli/internal/scan/scan.go:176-186`. Coercing to valid UTF-8 before `json.Valid` matches how the transcript is hashed/sent (`readSession`, line 208), so the client validates the exact bytes it will transmit — correct order. No case sends structurally invalid JSON or drops good JSON.

Worth recording for the record: `json.Valid` does **not** guarantee Postgres will accept the value as `jsonb`. A `\u0000` escape inside a JSON string passes `json.Valid` but Postgres rejects it on the `?::jsonb` cast (`SummaryRepository.java:37,80`). This is exactly why the server-side try/catch in finding #2 is load-bearing rather than paranoia — the two layers are complementary and both justified.

### 4. `readSummary` silently swallows genuine read errors (nit)
`cli/internal/scan/scan.go:177` treats a real I/O/permission error the same as file-absent (`err != nil` → `return ""`). This mirrors the "skip an unreadable transcript rather than fail" philosophy and is defensible for a best-effort summary, but unlike `readSession` (which returns the error) it discards it without distinguishing `os.IsNotExist`. Optional.

### 5. Test gaps — mostly fine (nit)
- Java `IngestorTest.aMalformedSummaryDoesNotCrashTheBatch...` genuinely fails without the change: without the catch, `s1`'s bad-jsonb `StoreUnavailableException` propagates out of `ingest`, so `s2` is never processed and no `IngestResult(2,0,0)` is returned. Good regression guard.
- Go `TestScanReadsAntigravitySummaryJSON` fails without the change (`Summary` would be `""`); `TestScanIgnoresMalformedSummaryJSON` meaningfully guards the `json.Valid` line. Both fine.
- Not tested: a summary that is valid JSON but a bare scalar (`null`, `123`) — `readSummary` returns it and the server stores jsonb `null`. Harmless, but no test pins the intended behavior. Optional.

### 1. Actionable — CLI silently drops summary-push failures
